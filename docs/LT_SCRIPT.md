## 1. 導入（1 分）

みなさんこんにちは。今日は mruby で Raspberry Pi 5 の GPU を使って顔認識をやってみた話をします。

まず、これを見てください。

```bash
$MRUBY -e 'GPU.init("shader"); a = GPU.array([1.0, 2.0, 3.0]); b = GPU.array([10.0, 20.0, 30.0]); puts GPU.add(a, b).head(3).inspect'
# => [11.0, 22.0, 33.0]
```

**mruby のたった 1 行で、GPU に計算させています。**

これを Raspberry Pi 5 で動かしています。3,000 円の GPU で、100 万個の数値の足し算を **5 ミリ秒** でやってくれます。

```bash
$MRUBY examples/demo.rb
```

```
GPU (Vulkan):  5.4 ms  ← 100万要素
CPU (mruby) :  133 ms
=> GPU is 24x faster!
```

GPU すごいですね。じゃあ次は、もっと実用的なことをやってみましょう。

## 2. カメラ顔認識デモ（2 分）

### OpenCV を使わずにカメラを動かす

ふつう、カメラで顔認識というと OpenCV を思い浮かべますよね。でも OpenCV は巨大で、Raspberry Pi には重すぎます。

今回は **OpenCV を一切使っていません。** 全部自前で書いています。

| やること | 普通の方法 | 今回の方法 |
|---------|-----------|-----------|
| カメラ映像の取得 | OpenCV の VideoCapture | **V4L2**（Linux カーネルの API を直接叩く） |
| 画像の色変換 | OpenCV の cvtColor | **自前の C 関数** (YUYV → RGB) |
| 顔の検出 | OpenCV の CascadeClassifier | **NCNN** (Tencent 製の軽量 AI 推論エンジン) |
| 画面への表示 | OpenCV の imshow | **SDL2** (ゲーム開発でおなじみ) |

### V4L2 って何？

**V4L2**（Video for Linux 2）は、Linux に標準で入っているカメラの窓口です。

USB カメラを Raspberry Pi に挿すと、Linux が `/dev/video0` というファイルを作ってくれます。
V4L2 は、このファイルを通じてカメラと会話するための仕組みです。

```
OpenCV を使う場合（百貨店で買い物）:

  mruby → OpenCV → V4L2 → カメラ
           ↑
        巨大なライブラリ（数百 MB）
        画像処理・機械学習・GUI... 全部入り
        「カメラ映像が欲しいだけなのに、百貨店ごと持ってくる」

今回の方法（産地直送）:

  mruby → V4L2 → カメラ
           ↑
        Linux に最初から入っている（追加インストール不要）
        「カメラさん、映像ください」と直接お願い
```

実際のコードはこれだけです:

```ruby
cam = Camera.open("/dev/video0", 640, 480)   # カメラを開く
frame = cam.capture                           # 1枚撮影
cam.close                                     # 閉じる
```

裏側の C 言語では、Linux の `ioctl` という命令でカメラに話しかけています。
「640×480 の映像をください」「YUYV 形式でお願いします」「準備できたら教えてね」
という会話を、150 行くらいの C で書いています。

ちなみに **YUYV** というのはカメラが返す映像の色の並び方です。
人間の目に見える RGB（赤緑青）とは違う形式なので、あとで RGB に変換しています。
これも OpenCV を使わず、自前の C 関数で変換しています。

### SDL2 って何？

カメラから映像を取れても、画面に映さなければ見えません。
ふつうは OpenCV の `imshow` で表示しますが、今回は OpenCV がないので **SDL2** を使っています。

**SDL2**（Simple DirectMedia Layer）は、ゲームを作るときによく使われるライブラリです。

```
有名な使用例:
  - Valve の Steam（ゲーム配信プラットフォーム）
  - Minecraft の一部バージョン
  - 多くのインディーゲーム
```

SDL2 は「ウィンドウを開いて、そこにピクセルを描く」のが得意です。
難しいことは考えず、**RGB のデータを渡すだけで画面に映してくれます。**

```ruby
disp = Display.open(640, 480, "face demo")   # ウィンドウを開く
disp.show(rgb, 640, 480)                      # RGB データを表示
disp.close                                    # 閉じる
```

顔を見つけたら、その位置に緑色の枠を描くのも SDL2 の仕事です。

```ruby
# 顔の位置 (x, y) に 幅w × 高さh の緑色の枠を描く
rgb = disp.draw_rect(rgb, 640, 480, x, y, w, h, 0, 255, 0)
```

まとめると、今回のシステムはこうなっています:

```
カメラ ──V4L2──→ 映像取得 ──C関数──→ 色変換 ──NCNN──→ 顔検出 ──SDL2──→ 画面表示
  ↑ 産地直送       ↑ 自前          ↑ AI エンジン     ↑ ゲーム用ライブラリ

  OpenCV なし。全部を軽量な部品で組み立てている。
  だから Raspberry Pi でも軽快に動く。
```

### 推論エンジン: NCNN

顔を見つける AI の頭脳には **NCNN** を使っています。

NCNN は中国 Tencent（WeChat の会社）が作った推論エンジンで、**WeChat のカメラ機能** で実際に使われています。スマホでサクサク動くように作られているので、Raspberry Pi でも軽快に動きます。

使っているモデルは **UltraFace-slim** — たった **1 MB** の超軽量モデルで、顔の位置を見つけてくれます。

### デモ実行

```bash
DISPLAY=:0 $MRUBY examples/face_demo.rb fast gpu
```

→ カメラ映像がウィンドウに映り、顔に緑の枠がつく。

**「mruby のスクリプトだけで、カメラから顔認識まで動きました。」**

## 3. GPU の意外な弱点（2 分）

### 「あれ、ちょっと遅くない？」

GPU モードだと **6 FPS** です。ヌルヌルとは言えない。

ここで CPU モードに切り替えてみます。

```bash
DISPLAY=:0 $MRUBY examples/face_demo.rb fast cpu
```

→ **30 FPS。** 明らかに滑らかに動く。

**CPU の方が 5 倍速い。** さっき 100 万個の計算で GPU が 24 倍速かったのに、なぜ？

### GPU と CPU の違い — 「工場」と「職人」

これ、**GPU と CPU の得意分野の違い** なんです。

```
GPU は「大きな工場」
  ├─ 作業員が 1,000 人いる（並列処理ユニット）
  ├─ 全員で同じ作業をすると超速い
  └─ でも「次の指示を出す → 届く → 作業する → 報告する」に時間がかかる
      ↑ これが "通信コスト"

CPU は「腕のいい職人」
  ├─ 4 人しかいないけど、一人ひとりが超優秀
  ├─ 手元にメモ帳がある（キャッシュ）からすぐ作業できる
  └─ 指示を出す手間がほぼゼロ
```

### 顔認識で何が起きているか

UltraFace は **100 層** のニューラルネットワークです。
GPU で動かすと、こうなります:

```
GPU での推論:
  「第1層を計算して」→ [通信] → 計算 → [通信] → 
  「第2層を計算して」→ [通信] → 計算 → [通信] → 
  ...
  「第100層を計算して」→ [通信] → 計算 → [通信]

  100 回も「お使い」に出している！
  1 回の通信に約 1 ミリ秒 → 100 回で 100 ミリ秒のロス
```

```
CPU での推論:
  第1層 → 第2層 → ... → 第100層

  全部手元で完結。通信コストゼロ。
  しかもモデルが 1MB と小さく、CPU の「メモ帳」（キャッシュ）に全部入る。
```

### 一言でまとめると

| | 100 万個の足し算 | 顔認識（100 層 × 小データ） |
|--|----------------|------------------------|
| GPU | **5ms（24 倍速い）** | 160ms（遅い） |
| CPU | 133ms | **6ms（5 倍速い）** |

**GPU は「大量のデータを一気にドーン」が得意。**
**「少ないデータを何回もやりとり」は苦手。**

顔認識は後者だったので、CPU が勝ちました。

## 4. じゃあ GPU はどう使う？（1 分）

### 今回の答え: 適材適所

```ruby
# GPU が得意: 画像処理（大データ × 1 回のドーン）
GPU.init("shader")
rgb = Camera.yuyv_to_rgb(yuyv, 640, 480)   # GPU で色変換

# CPU が得意: AI 推論（小データ × 100 回のやりとり）
detector = FaceDetector.new("models/ultraface-slim", use_gpu: false)
faces = detector.detect_rgb(rgb, 640, 480)  # CPU で推論
```

**mruby-gpu なら、`use_gpu: true` を `false` に変えるだけ。**
1 行の変更で GPU と CPU を使い分けられる。それが mruby-gpu の強みです。

### GPU をもっと活かすアイデア

実は、GPU が速くなるかもしれないアイデアがあります。

今は 1 枚ずつ写真を撮って 1 枚ずつ推論しています。
でもこうしたらどうでしょう？

```
現在:  撮る → 処理 → 撮る → 処理 → 撮る → 処理 ...

提案:  撮る撮る撮る撮る撮る撮る（300 枚まとめて録画）
       → GPU で 300 枚を一括前処理（ドーン！）
       → CPU で推論
```

**300 枚の写真をまとめて GPU に渡せば、「大量データを一気にドーン」のパターンになる。**

これから実装してみるところです。
うまくいけば、LT 会場の人数を「録画 10 秒 → 処理 2 秒 → この会場には XX 人います！」とできるかもしれません。

## 5. まとめ（30 秒）

今日お伝えしたかったこと:

1. **mruby からでも GPU は使える** — Raspberry Pi 5 の 3,000 円 GPU で 100 万要素を 5ms
2. **GPU は万能じゃない** — 顔認識では CPU の方が 5 倍速かった
3. **大事なのは適材適所** — GPU は「大量データ × 一括処理」、CPU は「小データ × 多段処理」
4. **mruby-gpu なら 1 行で切り替え可能** — `use_gpu: true/false` だけ

GPU と CPU、それぞれの得意分野を理解して使い分ける。
mruby-gpu がその橋渡しになれたら嬉しいです。

ありがとうございました。
