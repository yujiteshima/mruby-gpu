# パフォーマンス分析と改善アイデア

Raspberry Pi 5 / Logitech C920 (640×480) での実測データをもとに、  
ボトルネックと改善アイデアをまとめたドキュメントです。

---

## 実測プロファイル（10フレーム平均）

| 処理ステップ | 時間 | count モード合計への寄与 |
|------------|------|------------------------|
| V4L2 キャプチャ | 0.4ms | 0.4ms |
| YUYV → RGB 変換 | 2.5ms | 2.5ms |
| 全体推論 (640×480) | 163ms | 163ms |
| タイル推論 (320×240) × 5 | 161ms × 5 | 805ms |
| crop_rgb × 5 | 0.1ms × 5 | 0.5ms |
| **合計 (count モード)** | | **≈ 971ms (1.0 FPS)** |
| **合計 (fast モード)** | | **≈ 166ms (6.0 FPS)** |

### メモリアロケーション / フレーム

```
YUYV String        600 KB   (V4L2 → mruby String)
RGB String         900 KB   (yuyv_to_rgb)
tile ×5          1,125 KB   (crop_rgb ×5)
draw_rect コピー   900 KB × 顔の数
──────────────────────────────────
合計 (count, 1人)  3,525 KB / frame  → GC 頻発
```

## ボトルネック① NCNN 推論が支配的（count モードで 99%）

```
detect_full  163ms × 1 =  163ms
detect_tile  161ms × 5 =  805ms
─────────────────────────────────
合計                      968ms  ← 全体 971ms の 99%
```

### 改善アイデア

| # | アイデア | 期待効果 | 難易度 |
|---|---------|---------|--------|
| A | **NCNN Vulkan メモリプール有効化** — `ncnn::Option::use_local_pool_allocator = true` を設定。Vulkan メモリの都度確保を回避し、数十 ms 削減の可能性 | ★★☆ | 低 |
| B | **バッチ / 非同期推論** — 5タイルを逐次でなく非同期(`asyncForward`)で並列投入し GPU 稼働率を上げる | ★★★ | 高 |
| C | **INT8 量子化モデル** — `ncnn2table` + `ncnnoptimize` で UltraFace を INT8 量子化。V3D は INT8 をサポートしており 1.5〜2× の高速化が期待できる | ★★★ | 中 |
| D | **タイル数を 5 → 3 に削減** — 左上・右上・下中央の 3 タイルで画角の 9 割をカバー。count モードが ≈ 640ms に改善 | ★☆☆ | 低 |
| E | **推論スキップ** — 推論は 2 フレームに 1 回、表示は毎フレーム。前フレームの検出結果を表示に使い回す | ★☆☆ | 低 |

---

## ボトルネック② フレームあたり 3.5 MB のメモリコピー

毎フレーム大量の `mrb_str_new` が走り GC を誘発しています。

### 改善アイデア

| # | アイデア | 期待効果 | 難易度 |
|---|---------|---------|--------|
| F | **`draw_rect!` インプレース書き換え** — 現在の `draw_rect` は毎回フルフレーム(900 KB)をコピーして新 String を返す。元バッファを直接書き換える破壊的メソッドを追加すると、顔 N 人分のコピーが 0 になる | ★★☆ | 低 |
| G | **固定バッファの使い回し (`capture_rgb_into`)** — `Camera.yuyv_to_rgb` が毎フレーム 900 KB を新規確保。事前に確保した固定バッファに上書きする C メソッドを実装し GC 圧力を排除 | ★★☆ | 中 |
| H | **`crop_rgb` ゼロコピー化** — タイルを実際にコピーするのでなく、元 RGB バッファ上のオフセット + ストライドを `ncnn::Mat` に直接渡す。`ncnn::Mat` は外部バッファポインタを受け取れるため 1,125 KB のコピーが 0 になる | ★★★ | 中 |
| I | **V4L2 mmap の直接利用** — 現在 `capture` は カーネルの mmap バッファを `mrb_str_new` でコピー。NCNN・SDL2 に mmap ポインタを直接渡す C 内部パスを作り 600 KB のコピーを省く | ★★☆ | 高 |

## ボトルネック③ 直列パイプライン（CPU が GPU を待ちっぱなし）

```
現在:
  [capture 0.4ms] → [yuyv_to_rgb 2.5ms] → [detect 163ms] → ... → [display]
                                              ↑ GPU 推論中 CPU は完全アイドル

理想:
  フレームN:    [capture] → [yuyv_to_rgb] ─────────────────→ [display]
  フレームN+1:                             [detect N] → ...
```

### 改善アイデア

| # | アイデア | 期待効果 | 難易度 |
|---|---------|---------|--------|
| J | **スレッド分離（ダブルバッファリング）** — キャプチャ・推論・表示を別スレッドに分ける。mruby の C 拡張内では GIL が外れるため、NCNN 推論中にキャプチャを並行実行できる。理論上 fast モードでキャプチャ待ちがなくなる | ★★★ | 高 |
| K | **表示フレームレートと推論フレームレートを分離** — 推論結果を 1 フレームキャッシュし、表示は毎フレーム・推論は 1 フレームおきにすることで体感スムーズさが向上 | ★☆☆ | 低 |

---

## ボトルネック④ Ruby GC 圧力（一時オブジェクトの大量生成）

毎フレーム `sort_by`・`Array.new`・`Hash.new` 等で数十〜百の一時オブジェクトが生成されます。

### 改善アイデア

| # | アイデア | 期待効果 | 難易度 |
|---|---------|---------|--------|
| L | **`global_nms` を C++ に移す** — Ruby 実装の NMS は `sort_by` + `Array.new(size, false)` で毎フレーム多数のオブジェクトを生成。`detect_rgb` の内部で NMS まで完結させると GC 圧力がほぼゼロになる | ★★☆ | 低 |
| M | **Face を Hash から Struct に変更** — `{x:, y:, w:, h:, score:}` の Hash を毎フレーム生成している。`FaceBox = Struct.new(:x, :y, :w, :h, :score)` にするとオブジェクト生成コストが下がる | ★☆☆ | 低 |

## 優先度まとめ

即効性・難易度のバランスで優先度を整理します。

```
優先度 高（すぐ実装できて効果大）
  F: draw_rect! インプレース書き換え  → 900KB×顔数 のコピー削除
  L: global_nms を C++ に移す         → GC 圧力削減
  D: タイルを 5 → 3 に削減            → count モード 971ms → ~640ms

優先度 中（効果は大きいが実装コスト中）
  G: 固定バッファ capture_rgb_into    → 毎フレーム 900KB 確保を排除
  H: crop_rgb ゼロコピー化            → 毎フレーム 1,125KB コピーを排除
  A: NCNN メモリプール有効化          → 数十ms 削減

優先度 低（効果は最大だが実装コスト高）
  C: INT8 量子化モデル                → 推論 1.5〜2× 高速化（count で ~3 FPS）
  J: スレッド分離パイプライン         → キャプチャと推論を並行実行
  B: バッチ / 非同期推論              → GPU 稼働率向上
```
